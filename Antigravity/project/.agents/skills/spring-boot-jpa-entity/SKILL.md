# Spring Data JPA Entity Design

Detailed guidelines for designing database schemas and implementing robust Spring Data JPA entities.

## Overview

- Use when modeling a new domain and designing database tables from scratch.
- Focuses on generating safe and optimized entities, fundamentally preventing common anti-patterns such as the N+1 problem, circular references, and collection `NullPointerException`s.

## When to Use This Skill

- When starting a new feature that requires new database tables.
- When defining or modifying relationships (e.g., 1:N, N:M) between existing domain entities.
- When converting an ERD generated by the `database-erd-design` skill into actual Spring Boot code.

## How It Works

### Step 1: Define Base Auditing Entity

Create an abstract class `BaseEntity` that automatically manages `createdAt` and `updatedAt` fields using `@MappedSuperclass` and `@EntityListeners(AuditingEntityListener.class)`.
- **Note:** Remind the user that they must add the `@EnableJpaAuditing` annotation to their main application or configuration class for this to work.
- Strictly use the `jakarta.persistence.*` package instead of `javax.*`.

### Step 2: Implement Entity Classes

Create domain classes and add the `@Entity` annotation.
- Explicitly define the table name using `@Table(name = "table_names")` in plural snake_case format.
- You must add `@NoArgsConstructor(access = AccessLevel.PROTECTED)` to ensure the safe creation of JPA proxy objects.

### Step 3: Column and Primary Key (PK) Mapping

Define the primary key using `@Id` and `@GeneratedValue(strategy = GenerationType.IDENTITY)`.
- Specify constraints using `@Column`. Unless there is a special case like reserved word conflicts, omit hardcoding `@Column(name="...")` and follow Spring Boot's default naming strategy.
- Distinguish appropriately between `@Column(nullable = false)` for database schema constraints and `@NotNull` for application-level validation.

### Step 4: Define Associations and Collections

Carefully define associations (`@ManyToOne`, `@OneToMany`, `@OneToOne`).
- **CRITICAL:** You must explicitly specify `fetch = FetchType.LAZY` for all `@ManyToOne` and `@OneToOne` annotations.
- For `@OneToMany`, immediately initialize it with an empty collection upon declaration (e.g., `= new ArrayList<>()`).
- For bidirectional associations, write a 'Convenience Method' in the parent entity to safely synchronize the state of both objects.

## Examples

```java
// 1. Base Auditing Entity
@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @CreatedDate
    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @LastModifiedDate
    @Column(nullable = false)
    private LocalDateTime updatedAt;
}

// 2. Domain Entity
@Entity
@Table(name = "documents")
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Document extends BaseEntity {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 100)
    private String title;

    @Column(columnDefinition = "TEXT", nullable = false)
    private String content;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", nullable = false)
    private User author;

    @OneToMany(mappedBy = "document", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Comment> comments = new ArrayList<>();

    @Builder
    public Document(String title, String content, User author) {
        this.title = title;
        this.content = content;
        this.author = author;
    }

    public void updateContent(String title, String content) {
        this.title = title;
        this.content = content;
    }

    public void addComment(Comment comment) {
        this.comments.add(comment);
        comment.setDocument(this);
    }
}

```

## Best Practices

* **Utilize Business Methods:** Instead of blindly exposing `@Setter`, create meaningful business methods (e.g., `updateContent()`) that handle state changes to express the behavior of the domain.
* **Association Convenience Methods:** When adding a child entity in a bidirectional `@OneToMany` relationship, always implement a convenience method that updates the references of both objects.
* **Safe Builder Pattern:** Declare `@Builder` strictly on a custom **constructor**, not at the class level, to prevent ID or collection fields from being arbitrarily manipulated externally through the builder.
* **Consider Soft Delete:** For domains where data preservation is critical, consider applying logical deletion (Soft Delete) using Hibernate's `@SQLDelete` and `@SQLRestriction` (or `@Where`) instead of physical deletion.

## Common Pitfalls

* **Indiscriminate Use of Lombok:** **Never use** `@Data` or `@ToString` at the entity class level. They cause infinite loops (circular references) in bidirectional relationships. If equality comparison is needed, explicitly restrict it to the identifier field using `@EqualsAndHashCode(of = "id")`.
* **Leaving EAGER Loading Unchecked:** If the default fetch strategy (EAGER) of `@ManyToOne` is not explicitly changed to `LAZY`, it will cause unpredictable and massive N+1 queries, severely degrading performance.
