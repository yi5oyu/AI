# Spring Boot REST API Service Unit Test

Guidelines for writing ultra-fast unit tests that verify only the pure business logic inside the Service layer. This is achieved by completely avoiding the Spring framework (ApplicationContext) and replacing external dependencies (DB, external APIs, etc.) with fake mock objects.

## Overview

- Verifies that domain core rules—such as `if-else` branching, data manipulation, and exception throwing under specific conditions—work correctly inside the Service class, without database connections or Web layer (Controller) intervention.
- Since it does not load the Spring Boot framework, thousands of tests can be executed in just a few seconds. It is the lightest and most important testing skill with absolutely zero side effects.

## When to Use This Skill

- When verifying the mathematical/logical correctness of new business logic immediately after implementing it in a Service.
- When testing if an intended exception (e.g., `EntityNotFoundException`) is accurately thrown when attempting to query non-existent data in the DB.
- When rapidly and thoroughly testing all edge cases of complex domain logic that branches differently based on specific conditions.

## How It Works

### Step 1: Pure Mockito Environment

**NEVER use Spring-related annotations** (e.g., `@SpringBootTest`, `@WebMvcTest`) at the top of the test class.
- You **MUST use `@ExtendWith(MockitoExtension.class)`** to activate a pure Mockito environment.

### Step 2: Inject Mocks

Use **`@InjectMocks`** to create the actual Service class instance that is being tested.
- Use **`@Mock`** to inject fake objects (Mocks) for the Repositories or external components that the Service depends on.

### Step 3: BDD Stubbing & Execution

Strictly adhere to the BDD pattern (`// Given`, `// When`, `// Then`).
- `// Given`: Use **BDDMockito**'s `given(...).willReturn(...)` to set up the data (stubbing) that the fake Repository will return.
- `// When`: Call the method of the actual Service created with `@InjectMocks` to execute the logic.

### Step 4: Assertion & Verification

Strictly verify the results.
- **Normal Logic (`// Then`):** Use AssertJ's `assertThat()` to verify that the values of the returned DTO match expectations.
- **Exception Verification:** Use AssertJ's `assertThatThrownBy()` to verify that the intended business exception is accurately thrown under specific circumstances.
- **Interaction Verification:** Use `BDDMockito.then(...).should()` to verify if specific methods of the Repository were called. Use `ArgumentCaptor` when you need to verify the internal state of parameters passed to dependent objects.

## Examples

```java
@ExtendWith(MockitoExtension.class) // Use pure Mockito without Spring context (Ultra-fast)
class DocumentServiceUnitTest {

    @InjectMocks
    private DocumentService documentService; // The actual service object to test

    @Mock
    private DocumentRepository documentRepository; // Fake DB object the service depends on

    @Captor
    private ArgumentCaptor<Document> documentCaptor; // Captor to verify internal mapping logic

    @Test
    @DisplayName("Creates and saves a document entity and returns a response DTO on valid request")
    void createDocument_Success() {
        // Given
        DocumentCreateRequest request = new DocumentCreateRequest("Business Title", "Business Body");
        Document savedDocument = Document.builder()
                .title("Business Title")
                .content("Business Body")
                .build();
        
        // Inject values automatically generated by the DB (like IDs) using reflection
        ReflectionTestUtils.setField(savedDocument, "id", 1L); 

        // Stubbing: Return savedDocument when the Repository's save() is called
        given(documentRepository.save(any(Document.class))).willReturn(savedDocument);

        // When
        DocumentResponse response = documentService.createDocument(request);

        // Then: Verify the state of the returned DTO
        assertThat(response.id()).isEqualTo(1L);
        assertThat(response.title()).isEqualTo("Business Title");
        
        // Then: Verify interaction and the internal state of the entity passed to save()
        then(documentRepository).should(times(1)).save(documentCaptor.capture());
        Document capturedDocument = documentCaptor.getValue();
        assertThat(capturedDocument.getTitle()).isEqualTo("Business Title");
        assertThat(capturedDocument.getContent()).isEqualTo("Business Body");
    }

    @Test
    @DisplayName("Throws EntityNotFoundException when querying a non-existent document")
    void getDocument_Fail_WhenNotFound() {
        // Given
        Long invalidId = 999L;
        
        // Setup Repository to return an empty Optional
        given(documentRepository.findById(invalidId)).willReturn(Optional.empty());

        // When & Then: Verify the intended exception is accurately thrown
        assertThatThrownBy(() -> documentService.getDocument(invalidId))
                .isInstanceOf(EntityNotFoundException.class)
                .hasMessageContaining("Could not find the document");
                
        // Verify interaction: check if DB query was attempted exactly once
        then(documentRepository).should(times(1)).findById(invalidId);
    }
}

```

## Best Practices

* **Enforce Static Imports:** To maximize code readability, you **MUST use static imports** for AssertJ methods (`assertThat`, `assertThatThrownBy`) and Mockito methods (`any`, `times`, `given`, `then`).
* **Actively Utilize Reflection:** Using `ReflectionTestUtils.setField()` allows you to easily inject fake identifier (ID) values while preventing the design flaw of leaving unnecessary Setters open in entity classes just for testing.
* **Utilize ArgumentCaptor:** If the Service method converts a DTO to an Entity and then calls `save()`, use `@Captor` and `ArgumentCaptor` to catch the parameters passed to the fake repository and strictly verify the internal mapping logic.

## Common Pitfalls

* **Using @SpringBootTest (Critical):** Do not load the Spring context, which takes dozens of seconds, just to verify a single business logic. You MUST strictly maintain a pure Mockito environment (`@ExtendWith(MockitoExtension.class)`).
* **NPEs Due to Missing Stubbing:** Mock object methods return `null` (or empty collections) by default unless a `given()` setup is provided. To prevent `NullPointerException`s when the Service references these return values, meticulously set up the return values of all Mock methods in the execution path.
* **Overusing Interaction Verification:** Using `then().should()` on every single line tightly couples the test code to the internal implementation logic, making future refactoring difficult. Focus more on verifying the 'final outcome' (state return or specific exception throwing) of the business logic.
