# Spring Boot Optimized Dockerfile (Multi-stage)

Guidelines for creating ultra-lightweight containers by perfectly separating the source code compilation (Build) and execution (Run) environments within a single `Dockerfile`, ensuring that no unnecessary source code or build tools are left in the production environment.

## Overview

- **Stage 1 (Builder):** Uses a heavy image with the JDK (Java Development Kit) installed to download dependencies and build the application (generate the `.jar`).
- **Stage 2 (Runtime):** Uses a lightweight JRE (Java Runtime Environment) base image, copying only the completed `.jar` file from the Builder stage to execute it.
- Maximizes **Docker Layer Caching** by copying configuration files (`build.gradle`) that rarely change *before* the frequently changing source code (`src/`).

## When to Use This Skill

- When packaging a completed Spring Boot application into a container image for deployment to AWS ECS, EKS (Kubernetes), or a local `docker-compose` environment.
- When an existing Docker image is too large, causing slow deployment speeds or excessive disk usage (Optimization).
- When configuring automated container builds in a CI/CD pipeline (GitHub Actions, Jenkins).

## How It Works

### Step 1: Create `.dockerignore`

Create a `.dockerignore` file in the project root to prevent unnecessary files from being included in the Build Context, which would otherwise slow down the build.
- Included content: `.git`, `.gradle`, `build/`, `out/`, `*.md`

### Step 2: Write Stage 1 (Build)

Use a lightweight JDK base image like `eclipse-temurin:21-jdk-alpine`.
- Before copying the entire source code, copy ONLY `build.gradle`, `settings.gradle`, and `gradlew` to download dependencies (`.gradlew dependencies`). As long as `build.gradle` hasn't changed, this layer is cached, drastically reducing build time.
- Copy the source code next and build using the `bootJar` task.
- **Caution:** Delete the `-plain.jar` generated by Spring Boot 2.5+ to prevent copy conflicts in Stage 2.

### Step 3: Write Stage 2 (Run)

Use a JRE-only base image like `eclipse-temurin:21-jre-alpine`.
- **Security Setup:** Create and switch to a Non-root user with restricted privileges using `addgroup` and `adduser` so that even if the container is compromised, the attacker does not gain root access to the host system.
- Copy the single `.jar` file generated in Stage 1 (`builder`) and run the application via `ENTRYPOINT` configured with memory optimization flags.

## Examples

```dockerfile
# ==========================================
# Stage 1: Build Stage
# ==========================================
FROM eclipse-temurin:21-jdk-alpine AS builder

WORKDIR /build

# 1. Copy only Gradle wrapper and settings first (Maximize layer caching)
COPY build.gradle settings.gradle gradlew ./
COPY gradle ./gradle

# 2. Grant execution rights and download dependencies (Utilizes cache)
RUN chmod +x ./gradlew
RUN ./gradlew dependencies --no-daemon

# 3. Copy actual source code and build
COPY src ./src
RUN ./gradlew clean bootJar --no-daemon

# 4. Remove unnecessary plain.jar created in Spring Boot 2.5+ (Prevents COPY conflicts)
RUN rm -f build/libs/*-plain.jar

# ==========================================
# Stage 2: Runtime Stage
# ==========================================
FROM eclipse-temurin:21-jre-alpine

WORKDIR /app

# 1. Security: Create and switch to a restricted user (spring) without Root privileges
RUN addgroup -S spring && adduser -S spring -G spring
USER spring:spring

# 2. Copy only the executable jar built in the Builder stage
COPY --from=builder /build/build/libs/*.jar app.jar

# 3. Expose port (Documentation purpose)
EXPOSE 8080

# 4. Set Timezone and run the application
ENV TZ=Asia/Seoul

# MaxRAMPercentage=75.0: Prevents OOMKilled by using only 75% of the container's allocated memory for Heap
# /dev/urandom: Prevents delay (blocking) caused by random number generation during Tomcat startup
ENTRYPOINT ["java", \
            "-XX:MaxRAMPercentage=75.0", \
            "-Djava.security.egd=file:/dev/./urandom", \
            "-jar", "app.jar"]

```

## Best Practices

* **Container Memory Flags (`-XX:MaxRAMPercentage`):** In Kubernetes or Docker environments, you MUST use ratio-based settings instead of absolute values like `-Xmx`. This allows the JVM to flexibly adapt when container memory limits are changed.
* **Use Gradle Wrapper (`gradlew`):** Always use `gradlew` to guarantee that the build runs in the exact environment bound to the project, regardless of whether Gradle is installed (or its version) on the host machine or CI/CD server.

## Common Pitfalls

* **Copying Source Code All at Once (`COPY . .`) (Critical):** If you execute `COPY . .` at the top of the Dockerfile, the Docker cache breaks even if a single line of source code changes, forcing the re-download of hundreds of MBs of libraries every time. You MUST separate the copying of dependency files and source code.
* **Plain Jar Conflict (Critical):** When executing `COPY --from=builder .../*.jar`, if a `*-plain.jar` remains, the build will fail with a "multiple files matched" error. You must either remove the plain jar during the build stage or specify the exact file name.
* **Running as Root Account (Critical):** If a vulnerability occurs at container runtime while running as root, the host OS can be compromised. Always explicitly create a Non-root user and switch to it using the `USER` command.
* **Limitations of Alpine Base Images (musl libc):** Alpine is great for small sizes, but it lacks `glibc`. If you use C-based native libraries (e.g., image resizing, specific DB drivers), runtime errors may occur. In such cases, switch to `eclipse-temurin:21-jre-jammy` (Ubuntu-based).
